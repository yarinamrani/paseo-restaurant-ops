# Roadmap — מערכת תפעול לקבוצת המסעדות (פסאו · אומינו · טאלה)

מסמך עבודה מול המפרט המלא (יולי 2026). כל שלב נבנה בלי לשבור את מה שכבר רץ בפרודקשן.

## סטטוס שלבים

### ✅ שלב 1 — תשתית עסקים ואזורים (הושלם 2026-07-13)
- טבלאות `businesses` + `areas` עם RLS (קריאה למשתמש מאושר, ניהול לאדמין)
- `business_id`/`area_id` על tasks / admin_tasks / recurring_tasks, backfill מלא מהטקסט הישן
- עמודת `branch` הישנה נשארת מסונכרנת (תאימות לאחור + בוט וואטסאפ)
- כל הסינון/תגיות/דשבורד עובדים לפי business_id — שינוי שם עסק לא שובר כלום
- מסך "הגדרות" (אדמין): הוספה, שינוי שם, הסתרה/הפעלה של עסקים ואזורים
- שדה אזור (רשות) בטפסי תקלה ומשימה + תגית אזור על כרטיסים
- הבוט (wa_add_task) והמחולל של משימות חוזרות כותבים business_id

### ✅ שלב 2 — אחריות אמיתית והרשאות לפי עסק (הושלם 2026-07-13)
- `business_memberships` (משתמש↔עסק↔תפקיד) + פונקציית `has_business_access()`
- RLS פר-עסק על tasks / admin_tasks / recurring_tasks / task_history / businesses / areas — נבדק בפועל: משתמש בלי טאלה לא רואה נתוני טאלה
- `assignee_user_id` (אחראי ראשי FK) על כל הטבלאות; בטפסים בוחרים אחראי מרשימת הצוות (ערכי טקסט ישנים נתמכים)
- ניהול שיוך לעסקים במסך הצוות (צ'יפים ללחיצה); אישור משתמש חדש מעניק גישה לכל העסקים כברירת מחדל
- פילטר "שלי 👤" במשימות
- נדחה לשלב 3: משתתפים/עוקבים, "ממתין לאישור שלי" (תלוי בסטטוסים)

### ✅ שלב 3 — תגובות, Timeline, סטטוסים וסל מחזור (הושלם 2026-07-13)
- `item_comments` + `item_activity` (RLS דרך גישה לפריט האב; היומן immutable — נכתב רק ע"י טריגרים)
- טריגרים על tasks/admin_tasks מתעדים אוטומטית כל שינוי עם ערך ישן/חדש (כולל שינויים מהבוט)
- חלון פריט מלא (לחיצה על כותרת): סטטוס, תגובות, יומן פעילות, מחיקה
- 7 סטטוסים: פתוח / בטיפול / ממתין לספק / ממתין לאישור / מושהה / הושלם / בוטל
- Soft Delete + סל מחזור בהגדרות (שחזור / מחיקה לצמיתות לאדמין)
- נדחה: איחוד פיזי ל-work_items (הסיכון גבוה מהתועלת כרגע), Severity, Checklist, @אזכורים (שלב 4 עם התראות), חובת סיכום בסגירת סטטוס

### ✅ שלב 4 — התראות וקבצים מרובים (הושלם 2026-07-13)
- מרכז התראות: פעמון בכותרת עם מונה, פאנל, "סמן הכל כנקרא", עדכון **בזמן אמת** (supabase realtime)
- טריגרים: הוקצתה לך משימה/תקלה, שינוי סטטוס בפריט שלך, תגובה חדשה (לאחראי וליוצר) — בלי התראות לעצמך
- קבצים מרובים בחלון הפריט: תמונות (כולל HEIC), וידאו MP4/MOV, PDF/Office — עד 50MB, עם תיעוד ביומן ומחיקה (בעל הקובץ/אדמין)
- נדחה: Signed URLs (ה-bucket עדיין public — קישור בלתי-נחש אך נגיש למי שמחזיק בו), push/email, תמונת לפני/אחרי מובנית

### ✅ שלב 5 — תצוגות ודוחות (הושלם 2026-07-13)
- מסך משימות: 3 תצוגות (רשימה / קנבן / לוח שנה) עם זכירת ההעדפה
- קנבן: עמודות לפי סטטוס, גרירת כרטיס משנה סטטוס (דסקטופ); במובייל — דרך חלון הפריט
- לוח שנה חודשי לפי דד-ליין, סימון איחורים באדום
- דשבורד: KPI (באיחור / זמן טיפול ממוצע / % עמידה בזמנים), עומס פתוח לפי אחראי, ייצוא CSV (עברית, נפתח באקסל)
- נדחה: שמירת פילטרים אישיים, דוח תקלות חוזרות/ציוד

## אבטחה — יומן תיקונים
- 2026-07-14: **Batch תיקונים בטוחים (R1-phaseA / R4 / R6).** הוסרה מדיניות ה-listing הרחבה מהדלי (תמונות עדיין נטענות — HTTP 200 מאומת; רישום כל הדלי נחסם → []); `task_history` הוקפא ל-SELECT בלבד; בוטלה הרשאת EXECUTE ל-anon/PUBLIC על `generate_recurring_tasks` ו-`get_my_role` (הבוט `wa_add_task` ומדיניות ה-RLS לא נפגעו — מאומת). ללא שינוי קוד לקוח.
- 2026-07-14: **הוקשה הפרדת עסקים (Sprint 1 / משימה 1).** `has_business_access(null)` החזיר `true` = שורה ללא business_id נראתה חוצת-עסקים. תוקן: null→deny + CHECK constraints (NOT VALID) על tasks/admin_tasks/recurring_tasks המונעים יצירת פריט ללא עסק. אומת חי: null נדחה, שורה תקינה עוברת, גישה רגילה לא נפגעה. ללא שינוי קוד לקוח.
- פתוח (מ-Audit 2026-07-14): R1 **חלקי** — הדלי עדיין public (קובץ נגיש למי שמחזיק ב-URL המדויק); השלמה = דלי פרטי + Signed URLs (phase B, מסוכן, נדחה); R3 אין גיבוי/PITR; R5 role ב-membership לא נאכף (רק admin/staff); R7 הגדרות Auth (הגנת סיסמאות דלופות + אישור מייל) — הגדרות דשבורד, לא SQL.

## נושאים פתוחים (backlog)
- קבצים פרטיים + Signed URLs; Push לנייד; Email
- Severity + דגלי בטיחות לתקלות; Checklist; @אזכורים; חובת סיכום בסגירה
- איחוד פיזי ל-work_items; משתתפים ועוקבים; שמירת פילטרים; Realtime על רשימות

## החלטות ארכיטקטורה
- SPA (Vite+React+TS) על GitHub Pages + Supabase (Auth/DB/Storage/RLS) — נשאר; אין צורך ב-Next עד שנצטרך SSR
- שינויי DB רק דרך migrations (supabase); אין service_role בצד לקוח
- `branch` (טקסט) = עמודת legacy בלבד; מקור האמת הוא business_id
