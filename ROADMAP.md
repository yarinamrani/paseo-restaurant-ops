# Roadmap — מערכת תפעול לקבוצת המסעדות (פסאו · אומינו · טאלה)

מסמך עבודה מול המפרט המלא (יולי 2026). כל שלב נבנה בלי לשבור את מה שכבר רץ בפרודקשן.

## סטטוס שלבים

### ✅ שלב 1 — תשתית עסקים ואזורים (הושלם 2026-07-13)
- טבלאות `businesses` + `areas` עם RLS (קריאה למשתמש מאושר, ניהול לאדמין)
- `business_id`/`area_id` על tasks / admin_tasks / recurring_tasks, backfill מלא מהטקסט הישן
- עמודת `branch` הישנה נשארת מסונכרנת (תאימות לאחור + בוט וואטסאפ)
- כל הסינון/תגיות/דשבורד עובדים לפי business_id — שינוי שם עסק לא שובר כלום
- מסך "הגדרות" (אדמין): הוספה, שינוי שם, הסתרה/הפעלה של עסקים ואזורים
- שדה אזור (רשות) בטפסי תקלה ומשימה + תגית אזור על כרטיסים
- הבוט (wa_add_task) והמחולל של משימות חוזרות כותבים business_id

### ✅ שלב 2 — אחריות אמיתית והרשאות לפי עסק (הושלם 2026-07-13)
- `business_memberships` (משתמש↔עסק↔תפקיד) + פונקציית `has_business_access()`
- RLS פר-עסק על tasks / admin_tasks / recurring_tasks / task_history / businesses / areas — נבדק בפועל: משתמש בלי טאלה לא רואה נתוני טאלה
- `assignee_user_id` (אחראי ראשי FK) על כל הטבלאות; בטפסים בוחרים אחראי מרשימת הצוות (ערכי טקסט ישנים נתמכים)
- ניהול שיוך לעסקים במסך הצוות (צ'יפים ללחיצה); אישור משתמש חדש מעניק גישה לכל העסקים כברירת מחדל
- פילטר "שלי 👤" במשימות
- נדחה לשלב 3: משתתפים/עוקבים, "ממתין לאישור שלי" (תלוי בסטטוסים)

### ✅ שלב 3 — תגובות, Timeline, סטטוסים וסל מחזור (הושלם 2026-07-13)
- `item_comments` + `item_activity` (RLS דרך גישה לפריט האב; היומן immutable — נכתב רק ע"י טריגרים)
- טריגרים על tasks/admin_tasks מתעדים אוטומטית כל שינוי עם ערך ישן/חדש (כולל שינויים מהבוט)
- חלון פריט מלא (לחיצה על כותרת): סטטוס, תגובות, יומן פעילות, מחיקה
- 7 סטטוסים: פתוח / בטיפול / ממתין לספק / ממתין לאישור / מושהה / הושלם / בוטל
- Soft Delete + סל מחזור בהגדרות (שחזור / מחיקה לצמיתות לאדמין)
- נדחה: איחוד פיזי ל-work_items (הסיכון גבוה מהתועלת כרגע), Severity, Checklist, @אזכורים (שלב 4 עם התראות), חובת סיכום בסגירת סטטוס

### 🔜 שלב 4 — קבצים ותקשורת
- קבצים פרטיים + Signed URLs (כיום bucket ציבורי), ריבוי קבצים, וידאו MOV/MP4, מסמכים
- תמונת לפני/אחרי לתקלות; מרכז התראות פנימי; push/email
- Realtime (supabase channels) לעדכון חי

### 🔜 שלב 5 — דוחות ותצוגות
- Kanban, לוח שנה, שמירת פילטרים; דוחות SLA, תקלות חוזרות, עומסים; ייצוא CSV

## החלטות ארכיטקטורה
- SPA (Vite+React+TS) על GitHub Pages + Supabase (Auth/DB/Storage/RLS) — נשאר; אין צורך ב-Next עד שנצטרך SSR
- שינויי DB רק דרך migrations (supabase); אין service_role בצד לקוח
- `branch` (טקסט) = עמודת legacy בלבד; מקור האמת הוא business_id
